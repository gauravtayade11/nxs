How It Works

Every analysis request passes through the same pipeline regardless of which tool you use.

1. Input
file path · --stdin · --interactive · --pod (live fetch)
↓
2. Validation
max 512 KB · strip binary characters
↓
3. Secret detection
passive warning · --redact replaces 14+ patterns with [REDACTED]
↓
4. AI analysis
Groq → Anthropic → Mock fallback chain
↓
5. Result
{ severity, summary, rootCause, fixSteps, commands }
↓
6. Output
terminal · --output file.md · --notify slack · --fail-on exit code

After each analysis, nxs opens an interactive chat loop so you can ask follow-up questions with the original log and result held as context.

Configuration

Run the interactive setup wizard to configure your API keys:

bash
nxs config --setup

Or set keys directly from the command line:

bash
nxs config --set GROQ_API_KEY=gsk_...
nxs config --set ANTHROPIC_API_KEY=sk-ant-...
nxs config --set SLACK_WEBHOOK_URL=https://hooks.slack.com/services/...
nxs config --set NXS_API_KEY=your-server-secret

nxs config --get      # show saved keys (masked)
nxs config --reset    # clear all keys

Keys are stored in ~/.nxs/config.json with user-only file permissions. Environment variables always take precedence over the config file.

nxs tries providers in priority order and falls back automatically on rate limits or network errors.

Premium

pattern-match

Input Modes

All analysis commands accept log input in four ways.

File path

bash
nxs k8s debug pod-error.log

Stdin — pipe from any command

bash
kubectl logs my-pod --previous | nxs k8s debug --stdin
docker build . 2>&1 | nxs devops analyze --stdin

Interactive — paste directly in the terminal

bash
nxs k8s debug --interactive
# Paste your log, then type END on its own line to submit

Live cluster fetch (k8s tool only)

bash
# Auto-fetch logs + describe for a single pod
nxs k8s debug --pod api-server -n production

# Auto-fetch from all pods in a deployment
nxs k8s debug --deployment my-app -n staging

The combined block sent to the AI looks like:

text
=== POD: api-server-7d9f — LOGS ===
[kubectl logs output]
=== POD: api-server-7d9f — DESCRIBE ===
[kubectl describe output]

Output & Integration Flags

These flags are available on every analyze, debug, and diagnose command.

FlagDescription
--redactScrub secrets from input before sending to AI. Replaces 14+ patterns (AWS keys, tokens, PEM blocks, passwords) with [REDACTED].
-j, --jsonPrint raw JSON result and exit. Skips the interactive chat loop. For scripting and CI.
--no-chatPrint result and exit without prompting for follow-up questions.
-o, --output <file>Write a full markdown report to a file.
--notify slackPost result to Slack. Requires SLACK_WEBHOOK_URL. Color-coded by severity.
--fail-on <severity>Exit with code 1 if result matches critical or warning. Use as a CI gate.
-s, --stdinRead log from stdin.
--interactivePrompt for pasted input. End with END on its own line.

CI gate example

bash
kubectl logs my-pod --previous \
  | nxs k8s debug --stdin \
      --fail-on critical \
      --notify slack \
      --no-chat

# Exit 0 = info or warning
# Exit 1 = critical  →  fails the pipeline step
⚠

The exit code is lost when you capture output in a subshell: result=$(nxs … --fail-on critical). Run nxs directly so it can set the process exit code.

History & Reports

nxs saves the last 50 analyses to ~/.nxs/history.json and rotates automatically when the limit is reached.

bash
nxs history                        # 20 most recent
nxs history -n 50                  # up to 50 entries
nxs history --search "crash"       # filter by keyword
nxs history --json                 # raw JSON
nxs history --clear                # delete all history

Digests

bash
nxs report                         # last 7 days
nxs report --days 30               # last 30 days
nxs report --output report.md      # save to file
nxs report --notify slack          # post to Slack
nxs report --json                  # severity counts as JSON

GitHub Actions Integration

Get an AI diagnosis in Slack automatically whenever a build fails.

yaml

  notify-on-failure:
    needs: [build, test]
    if: failure()
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install
      - name: Diagnose failure

          {
            echo "Workflow: ${{ github.workflow }}"

Security

Secrets in logs

nxs applies two layers of protection on every run:

  1. Passive warning — scans input and prints a warning if secrets are detected. Never blocks the analysis.
  2. Active redaction — pass --redact to replace detected patterns with [REDACTED] before the log reaches any AI provider.
CategoryExamples
AWS credentialsAKIA… access keys, secret keys, session tokens
Auth tokensBearer …, Basic auth, JWT (eyJ…)
Private keysPEM blocks (-----BEGIN … KEY-----)
PasswordsDB connection strings, PASSWORD=… env vars
GCPService account JSON, OAuth tokens
AzureSAS tokens, AccountKey=…
Developerghp_… GitHub tokens, npm_…, Groq & Anthropic keys

Data storage

  • Raw log input is never written to disk
  • History stores only: severity, summary, first 200 characters of input, full result object
  • Config stored at ~/.nxs/config.json with user-only file permissions

API server auth

  • All endpoints except /health and /webhook/* require the X-Api-Key header
  • Every request is logged with timestamp, method, path, status code, and latency

Developer Guide — Adding a Tool

Each tool is a self-contained module in cli/tools/. Adding a new one takes two steps.

Step 1 — Create the module

javascript
// cli/tools/mytool.js
import { runAnalyze, runHistory } from '../core/runner.js';

const MODULE = 'mytool';

// Must instruct the model to return ONLY valid JSON — no prose, no fences.
const SYSTEM_PROMPT = `You are an expert in [domain].
Analyze the provided log and return ONLY valid JSON:
{
  "tool": "mytool",
  "severity": "critical | warning | info",
  "summary": "One sentence.",
  "rootCause": "Numbered list.",
  "fixSteps": "Numbered list.",
  "commands": "Shell commands, one per line."
}`;

// Covers common patterns when no API key is available.
function mockFn(log) {
  if (log.includes('connection refused')) {
    return { tool: 'mytool', severity: 'critical',
      summary: 'Service unreachable.', rootCause: '...',
      fixSteps: '...', commands: 'systemctl status myservice' };
  }
  return { tool: 'mytool', severity: 'info', summary: 'No issues.',
    rootCause: '—', fixSteps: '—', commands: '—' };
}

export function registerMyTool(program) {
  const cmd = program.command('mytool').description('Analyze [domain] errors');

  cmd.command('analyze [file]')
    .option('-s, --stdin')
    .option('-j, --json')
    .option('--redact')
    .option('-o, --output <file>')
    .option('--notify <target>')
    .option('--fail-on <severity>')
    .action(async (file, opts) => {
      await runAnalyze(MODULE, SYSTEM_PROMPT, mockFn, file, opts);
    });

  cmd.command('history')
    .option('-n, --limit <n>', 'Entries to show', '20')
    .option('--search <term>')
    .option('-j, --json')
    .option('--clear')
    .action(async (opts) => {
      await runHistory(MODULE, opts);
    });
}

Step 2 — Register in cli/index.js

javascript
import { registerMyTool } from './tools/mytool.js';

registerMyTool(program); // alongside the other registerXxx() calls
✦

You get --redact, --notify, --fail-on, --output, and --json for free from runAnalyze. Never re-implement them in your module.

⚠

Never write raw log text to disk inside a tool module. Only the result object and a 200-character preview are persisted, and the runner handles that automatically.

Troubleshooting

Analysis always shows mock responses

nxs falls back to mock mode when no valid API key is found.

bash
# Check saved keys
nxs config --get

# Verify the Groq key works
curl -s https://api.groq.com/openai/v1/models \
  -H "Authorization: Bearer $GROQ_API_KEY" | head -c 200

kubectl commands fail inside a tool

bash
which kubectl
kubectl config current-context
kubectl auth can-i get pods -n production
kubectl auth can-i get pods/log -n production

Prometheus alerts not reaching nxs

bash
# 1. Check pod status
kubectl get pods -n monitoring -l app=nxs-server
kubectl logs -n monitoring deploy/nxs-server --tail=50

# 2. Test webhook directly
kubectl port-forward -n monitoring svc/nxs-server 4000:4000 &
curl -X POST http://localhost:4000/webhook/alertmanager \
  -H "Content-Type: application/json" \
  -d '{"alerts":[{"labels":{"alertname":"PodCrashLooping",
       "severity":"critical","namespace":"default","pod":"my-pod"},
       "annotations":{"summary":"Pod crash looping"}}]}'

PodCrashLooping alert never fires

The expression requires a non-zero restart rate over the past 5 minutes. Test with a demo pod:

bash
kubectl apply -f demo/k8s/crash-loop.yaml
# Wait 2-3 minutes, then check Prometheus alerts UI

--fail-on not producing exit code 1 in CI

bash
# Wrong — exit code is swallowed by the subshell
result=$(nxs k8s debug pod.log --fail-on critical)

# Correct — let nxs exit the process directly
nxs k8s debug pod.log --fail-on critical --no-chat

History not updating

bash
ls -la ~/.nxs/
nxs history --clear    # reset if needed
