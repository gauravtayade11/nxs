nxs — AI DevOps
Debugging CLI

Paste any broken log — Kubernetes, Docker, Terraform, cloud, CI/CD, databases — and get an instant root cause and copy-paste fix commands. Works free with Groq or without any API key at all.

bash — nxs demo
$ kubectl logs api-pod --previous | nxs k8s debug --stdin
Analyzing… ████████████████ done
☸ KUBERNETES DETECTED — critical
📋 SUMMARY
Pod api-pod is crash-looping due to an OOMKilled exit.
🔍 ROOT CAUSE
1. Container memory limit is 256Mi
2. App allocates ~310Mi at startup
3. Kernel OOM killer terminates the process
💻 COMMANDS
kubectl set resources deployment/api --limits=memory=512Mi
kubectl rollout status deployment/api

⚙How It Works

Every analysis — regardless of tool — passes through the same pipeline.

1
Input
File path · --stdin · --interactive · --pod (live kubectl fetch)
2
Validation
Max 512 KB · strip binary characters
3
Secret detection
Passive warning if secrets found · --redact replaces 14+ patterns with [REDACTED]
4
AI analysis
Groq → Anthropic → Mock fallback — always returns a result
5
Structured result
{ severity, summary, rootCause, fixSteps, commands }
6
Output
Terminal · --output file.md · --notify slack · --fail-on exit code · interactive Q&A

◆AI Providers

nxs tries providers in order and falls back automatically on rate limits or network errors.

Priority 1
Groq
llama-3.3-70b-versatile
Free tier. Input truncated to 8 000 chars. Get a key at console.groq.com.
Priority 2
Anthropic
claude-opus-4-6
Extended thinking enabled. Automatic fallback when Groq is unavailable.
Fallback
Mock mode
pattern-match
No API key or network required. Tuned for the most frequent error patterns.
✦

Mock mode is not a degraded experience. It returns actionable output for common Kubernetes, Docker, and CI errors with no internet connection.

⚑Output & Integration Flags

Available on every analyze, debug, and diagnose command.

FlagDescription
--redactScrub 14+ secret patterns from input before sending to AI
-j, --jsonRaw JSON output. Skips chat loop. Use in scripts and CI.
--no-chatPrint result and exit without interactive Q&A
-o, --output <file>Write full markdown report to a file
--notify slackPost result to Slack (requires SLACK_WEBHOOK_URL)
--fail-on <sev>Exit code 1 if severity matches critical or warning
-s, --stdinRead log from stdin
--interactivePrompt for pasted input, end with END

CI gate example

bash
kubectl logs my-pod --previous \
  | nxs k8s debug --stdin \
      --fail-on critical \
      --notify slack \
      --no-chat

# exit 0 = info/warning  |  exit 1 = critical
⚠

The exit code is lost when you capture output in a subshell: result=$(nxs … --fail-on critical). Run nxs directly so it can set the process exit code.

◷History & Reports

nxs history --json           # raw JSON
nxs history --clear

nxs report --output report.md

⬡Developer Guide

Each tool is a self-contained module in cli/tools/. Adding one takes two steps.

javascript — cli/tools/mytool.js
// cli/tools/mytool.js
import { runAnalyze, runHistory } from '../core/runner.js';

const MODULE = 'mytool';

// Must return ONLY valid JSON — no prose, no markdown fences
const SYSTEM_PROMPT = `You are an expert in [domain].
Return ONLY valid JSON:
{ "tool":"mytool", "severity":"critical|warning|info",
  "summary":"...", "rootCause":"...",
  "fixSteps":"...", "commands":"..." }`;

function mockFn(log) {
  if (log.includes('connection refused'))
    return { tool:'mytool', severity:'critical', summary:'...',
              rootCause:'...', fixSteps:'...', commands:'systemctl status svc' };
  return { tool:'mytool', severity:'info', summary:'No issues.',
            rootCause:'—', fixSteps:'—', commands:'—' };
}

export function registerMyTool(program) {
  const cmd = program.command('mytool');
  cmd.command('analyze [file]')
    .option('-s, --stdin').option('-j, --json')
    .option('--redact').option('-o, --output <f>')
    .option('--notify <t>').option('--fail-on <s>')
    .action(async (file, opts) => {
      await runAnalyze(MODULE, SYSTEM_PROMPT, mockFn, file, opts);
    });
  cmd.command('history')
    .option('-n, --limit <n>', '', '20').option('--search <t>')
    .option('-j, --json').option('--clear')
    .action(async (opts) => { await runHistory(MODULE, opts); });
}
javascript — cli/index.js
import { registerMyTool } from './tools/mytool.js';
registerMyTool(program);
✦

--redact, --notify, --fail-on, --output, and --json come free from runAnalyze. Never re-implement them.

⛔

Never write raw log text to disk inside a tool module. History persistence is handled by the runner automatically.

⚕Troubleshooting

Always getting mock responses

bash
nxs config --get
curl -s https://api.groq.com/openai/v1/models \
  -H "Authorization: Bearer $GROQ_API_KEY" | head -c 200

kubectl fails inside a tool

bash
which kubectl
kubectl config current-context
kubectl auth can-i get pods/log -n production

Prometheus alerts not reaching nxs

bash
kubectl get pods -n monitoring -l app=nxs-server
kubectl logs -n monitoring deploy/nxs-server --tail=50

# Test the webhook directly
kubectl port-forward -n monitoring svc/nxs-server 4000:4000 &
curl -X POST http://localhost:4000/webhook/alertmanager \
  -H "Content-Type: application/json" \
  -d '{"alerts":[{"labels":{"alertname":"PodCrashLooping",
    "severity":"critical","namespace":"default","pod":"my-pod"},
    "annotations":{"summary":"crash looping"}}]}'

--fail-on not working in CI

bash
# Wrong — exit code swallowed by subshell
result=$(nxs k8s debug pod.log --fail-on critical)

# Correct
nxs k8s debug pod.log --fail-on critical --no-chat

PodCrashLooping alert never fires

The expression needs a non-zero restart rate over 5 minutes. Test with a demo pod:

bash
kubectl apply -f demo/k8s/crash-loop.yaml
# Wait 2-3 minutes, then check Prometheus /alerts
